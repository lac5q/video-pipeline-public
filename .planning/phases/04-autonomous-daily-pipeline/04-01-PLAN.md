1. **Create `lib/discord.js`** -- Discord webhook notification helper:
   - Export `sendDiscord(message, options)` function
   - Uses native `https` module (match existing codebase pattern from lib/consent.js -- no axios/node-fetch)
   - `message`: plain text content
   - `options.embed`: optional embed object with `title`, `description`, `color` (integer), `fields` array
   - Reads `DISCORD_WEBHOOK_URL` from `process.env`
   - If `DISCORD_WEBHOOK_URL` not set: log warning to stderr, return silently (non-fatal)
   - POST to webhook URL with `{"content": message, "embeds": [embed]}` if embed provided
   - Export `formatRunSummary(stats)` helper that builds a Discord embed:

<task type="auto">
  <name>Task 2: Enhance daily-pipeline.sh with consecutive circuit breaker, checkpoint validation, dual video build, and Discord notifications</name>
  <files>scripts/daily-pipeline.sh</files>
  <action>
Rewrite `scripts/daily-pipeline.sh` to incorporate all Phase 4 requirements. Keep existing structure and argument parsing but enhance with:

1. **Consecutive failure circuit breaker** (replaces total error count):
   - Track `consecutive_errors` variable, reset to 0 on each successful order
   - Trip at `MAX_CONSECUTIVE_ERRORS` (default 3, configurable via `--max-errors` arg or env var)
   - Keep `total_errors` for reporting but circuit breaker uses consecutive count
   - On trip: call `send_discord_notification "circuit_breaker"` before exiting

2. **Discord notification wrapper function** -- `send_discord_notification()`:
   - Calls `node -e "require('./lib/discord').sendDiscord(...)"` with the run summary stats
   - Two modes: "summary" (end of run) and "circuit_breaker" (on trip)
   - For summary: pass `{ runId, brandsProcessed, ordersAttempted, ordersSucceeded, ordersFailed, ordersSkipped, errors, duration }`
   - For circuit breaker: call `sendCircuitBreaker(consecutiveErrors, lastErrors)`
   - Track `RUN_START_TIME=$(date +%s)` at pipeline start, compute duration at end

3. **Enhanced checkpoint validation** -- replace loose existence checks with formal validation:
   - **Checkpoint 1 (Assets):** After download, verify `${WORKSPACE}/photos/` is non-empty AND every file in photos/ is > 0 bytes. Use: `find "${WORKSPACE}/photos/" -type f -size 0 | head -1` -- if any zero-byte file found, fail the checkpoint.
   - **Checkpoint 2 (Mockups):** After mockup generation, verify `v11_*.png` count > 0 AND each file > 0 bytes.
   - **Checkpoint 3 (Staging):** After Gemini staging, verify `.raw` backup files exist (indicates staging ran). Count staged mockups.
   - **Checkpoint 4 (Video):** Delegate to `verify-video.sh` (already comprehensive -- checks 1080x1920, h264, ~30fps, size > 0).
   - **Checkpoint 5 (Upload):** After Drive upload, verify the upload command returned exit 0 (already checked).
   - On any checkpoint failure: log specific checkpoint name + failure reason, increment errors, skip order.

4. **Dual video build** (match batch-produce.sh pattern):
   - Replace single `produce-video.sh` call with the `build-video.sh` pattern from batch-produce.sh
   - Always build standard reel: `build-video.sh "$BRAND_CONFIG" "$WORKSPACE" "$PRODUCTS_CONFIG" "reels"`
   - Detect reaction video (check for `${ORDER_ID}.{mov,mp4,MOV,MP4}` or `reaction.{mov,mp4,MOV,MP4}`)
   - If reaction exists, also build UGC reel: `build-video.sh "$BRAND_CONFIG" "$WORKSPACE" "$PRODUCTS_CONFIG" "ugc"`
   - UGC build failure is non-fatal (standard reel still available)
   - Verify BOTH output videos with verify-video.sh

5. **Failed order DB tracking**:
   - On order failure: update `production_status = 'failed'` in orders table with error detail
   - On order success: update `production_status = 'complete'` with video path
   - Track failed order IDs in an array for the run summary

6. **Run state persistence**:
   - At pipeline start: INSERT into `daily_runs` table with run_id and status='running'
   - At pipeline end: UPDATE with completed_at, final counts, status='complete' or 'failed' or 'circuit_breaker'
   - Use node one-liners to interact with DB (same pattern as existing line 286-292)

7. **Keep all existing features intact:**
   - `--dry-run`, `--skip-import`, `--brand`, `--limit`, `--max-errors`, `--help`
   - Log file rotation to `logs/daily-{timestamp}.log`
   - `tee -a` pattern for stdout + log file
   - Brand config validation
   - Import tracking sheets stage

8. **Add `--requeue` flag:**
   - When passed, reset all orders with `production_status = 'failed'` back to `pending` for this brand, then proceed with normal pipeline
   - Log how many orders were re-queued
  </action>
